## Audit: why nothing was merging

The cluster runner was running every 15 s on the user's session. Every
run was producing exactly zero merges. The summary block looked like:

```
abstains:
   1 hearing_aid    below_min_total_weight:0.40/0.55
   1485 apple_device   below_min_total_weight:0.25/0.60
   10 mac             below_min_total_weight:0.25/0.60
   153 airtag         missing_eventually:rotation_cohort
```

Three concurrent gaps:

1. **`apple_device` (1485 pairs)** — falls back to the `default`
   profile, which lists six weighted signals summing to 1.00. Only
   two are implemented (`service_uuid_match` 0.30 +
   `mfg_data_prefix` 0.25 = 0.55 max). Most apple_device pairs only
   had mfg_data scoring (0.25 weight) since they're RPAs without
   UUID16 history. 0.25 < 0.60 `min_total_weight` → silent abstain
   on every pair.

2. **`hearing_aid` (1 pair, the C82 13418 vs 13677)** —
   `service_uuid_match` returns 0.0 (no UUID overlap) at weight
   0.40. mfg_data_prefix abstains because the C82 doesn't broadcast
   manufacturer data at all. 0.40 < 0.55 → abstain.

3. **`airtag` (153 pairs)** — profile lists
   `required_eventually = ["rotation_cohort"]`. The rotation_cohort
   signal *exists* but reads observation timestamps from
   `ctx.cache["observations"]` — a cache populated only by the
   synthetic-data tests, never in production. So `applies_to`
   returns False for every pair, the signal lands in
   `missing_eventually`, and the aggregator returns None.

   This is a 1-line fix (add a DB-loading path that queries
   `packets`) but it's not yet in scope for this deep dive — flagged
   as the highest-priority follow-up below.

## Signal #1: apple_continuity

The profiles already referenced this signal as `required_for_merge`
for airpods (which made every airpods pair abort with
`missing_required_for_merge:apple_continuity`). Implementing it
unlocks airpods + improves apple_device and iphone immediately.

### Protocol

Apple stuffs Continuity TLVs into manufacturer-specific advertising
data (AD type 0xFF) under their company id 0x004C:

```
[0x4C, 0x00, <type><length><payload>, <type><length><payload>, ...]
```

Common types observed in the captured data:

| Type | Name | What it carries |
|---|---|---|
| 0x07 | ProximityPairing | AirPods + Beats: model id (bytes 1-2), state, battery |
| 0x09 | AirPlaySource | Speakers / Apple TV broadcasting available content |
| 0x0C | Handoff | User-activity continuation (encrypted, rotates with RPA) |
| 0x10 | NearbyInfo | Action type (top 4 bits) + status flags + 4-byte rotating auth |
| 0x12 | Pairing | Short state code OR Find-My anchor (variant by length) |

### Scoring rationale

Long Continuity payloads (≥ 8 bytes) embed encrypted session keys,
sequence counters, or stable-byte identifiers that an unrelated device
is extremely unlikely to emit by coincidence. So a byte-for-byte match
between two devices is near-certainty same physical advertiser
captured under different RPAs.

Short payloads (e.g. 0x12 with 2-byte state code) are state enums
shared by many devices of the same model — too generic to fingerprint
as same-device. Treated as "common type" evidence only.

Disjoint Continuity vocabularies = different Apple roles entirely
(e.g. iPhone vs AirPods). Mild active negative; aggregator uses it as
opposing evidence.

| Pair situation | Score |
|---|---|
| Long-payload exact match (≥ 8 bytes) | **1.0** |
| Common TLV types only (no exact match) | **0.4 × jaccard** |
| Disjoint type vocabularies | **−0.3** |
| One/both have no Continuity at all | **None** (abstain) |

### Live-DB validation

Run against the user's session, recent 1-hour window:

```
71 devices  →  64 clusters  (13 merges, all score 1.0)
```

One of those clusters: **6 RPAs collapsed to 1 physical Apple
device**. Verified ground truth:

- All 6 device rows had RPA-typed addresses
- All 6 had lifetimes between 651 and 788 s (10:51 to 13:08) — exactly
  Apple's standard 15-minute RPA rotation interval
- Each pair shared 18-48 long Continuity payloads with its immediate
  neighbors in time, including sequential Handoff session counters
  (0x03, 0x04, 0x05, 0x06, 0x07, 0x08, 0x09, 0x0A …) captured under
  both RPAs across the rotation handover

### Scaling characteristics

| Window | Devices | Pairs | Merges | Clusters absorbed | Elapsed |
|---|---|---|---|---|---|
| 1 h | 71 | 1.4 K | 13 | 7 | 39 ms |
| 6 h | 473 | 58 K | 24 | 18 | 1.2 s |
| 24 h | 843 | 196 K | 46 | 31 | 4.4 s |
| all-time | 3 709 | 3.2 M | 54 | 39 | 33 s |

Long-window collapse rate is intentionally low: Continuity payloads
rotate every ~15 min, so RPA pairs separated by hours don't share
exact long payloads. The signal is high-precision, narrow-window. To
extend coverage across rotation epochs we need either a
**stable-byte** signal (catches same-AirPods-different-rotation via
model bytes) or **co-session correlation** (same physical device shows
up in same sniffer sessions with consistent packet ratios).

### Performance follow-up

At the all-time window the runner does ~6.4 M DB lookups (3.2 M pairs
× 2 device fingerprints per pair). Caching each device's parsed
fingerprint in `ctx.cache` once at run start — the framework already
provides this path — would drop runtime by an order of magnitude.
Recommend implementing as part of any future signal that does similar
per-device DB reads (`stable_byte_match`, `bis_access_address_match`,
…) so they all share the cache rather than each re-querying.

## Enriched Continuity parser (`_continuity_protocol.py`)

A separate utility module that any signal can use to get rich TLV
information rather than raw bytes. The parser:

- Names every type using a catalog drawn from FuriousMAC + the
  Celosia/Cunche "Discontinued Privacy" paper
- Flags per-type stable-byte prefix counts (so signals can match
  across rotation epochs by stable bytes only)
- Decodes specific types into structured fields:
  - **0x07 ProximityPairing**: AirPods/Beats model lookup
    (e.g. `0x1420` → "AirPods Pro 2nd gen"), battery percentage from
    nibble pairs, state byte
  - **0x10 NearbyInfo**: action code (top 4 bits) named per iOS RE
    sources (e.g. 0x4 = TVTransferAuthority, 0x7 = ScreenSharing),
    status-bit decoding (WiFi-on, AirPods-connected, authenticated)
  - **0x12 Pairing**: variant detection (state-code vs Find-My
    anchor vs short-auth) based on payload length

Coverage caveat: every type definition is best-effort. Apple revises
encodings between iOS releases; treat decoded fields as advisory. The
TLV-level parsing (types, lengths, stable-byte prefix counts) is
spec-grounded and solid.

The model catalog has 19 known AirPods/Beats variants today. The
inventory pass (next section) found 7 devices broadcasting model
`0x2720` which isn't in the catalog — likely AirPods 4th gen or a
recent variant; left as `unknown (2720)` until confirmed.

## Recommended next signals (in priority order)

### 1. `rotation_cohort` DB-loading fix [DONE in this PR; gated on §1c below]

The signal had been abstaining on every airtag pair in production
because `_observations_on_sniffer` read from `ctx.cache["observations"]`
— never populated outside of synthetic-data tests. Added a
DB-backed lazy-load path that queries the `packets` table on first
read and caches per-device.

The fix is mechanically correct, but discovered during live
validation that **the `packets` table is currently empty** (see
§1c). Until packet writes are enabled, the airtag pairs continue
to abstain on `missing_eventually:rotation_cohort` because
`applies_to` (correctly) returns False when the device has no
packet history.

### 1b. `rotation_cohort` profile-parameter inconsistency [BLOCKING airtag merges, ~1 hour]

The airtag profile sets:

```toml
[airtag.params.rotation_cohort]
expected_rotation = 900.0   # AirTag rotation interval (15 min)
window_max        = 60.0    # plausible handoff-gap window
```

But the signal's score formula is:

```python
if best_gap > p.window_max: return 0.0
delta = abs(best_gap - p.expected_rotation)
return max(0.0, 1.0 - delta / p.expected_rotation)
```

With `window_max=60` and `expected_rotation=900`, the
`delta`-from-`expected` calculation runs only when `best_gap ≤ 60`,
so `delta ≥ 840` always, and the returned score is ≤ 0.067. The
"expected_rotation = 900" parameter is therefore **unreachable** —
no parameter combination of the airtag profile can produce a
high-score handoff under the existing math.

Two ways to fix:

1. Rename and reinterpret. `expected_rotation` is misnomered;
   what the score really wants is "expected handoff gap." Call it
   `expected_handoff_gap_s` and set it to the realistic value
   (~10-30 s during the rotation overlap). Keep
   `expected_rotation` as a separate metadata field if it's needed
   anywhere else.

2. Change the score formula. Compare `best_gap` against a fraction
   of `window_max` instead, or use a Gaussian centered on
   `expected_rotation` with stddev related to `window_max`.

Either fix is small. Recommendation: rename. Cleaner semantics and
the math becomes obvious.

### 1c. `LiveIngest` doesn't write to the `packets` table [HIGH, ~30 min]

Discovered during live validation of the rotation_cohort fix —
`SELECT COUNT(*) FROM packets` returns **0** despite ~3 days of
captures and 91 M+ rows across other tables. The reason:

```python
# capture/live_ingest.py:129
self._ctx = IngestContext(session_id=sess.id)
```

`IngestContext.keep_packets` defaults to `False`, and the canvas
never overrides it. The packets-table writer in
`ingest/pipeline.py:227` is gated behind `ctx.keep_packets`, so
nothing ever lands in the table.

This blocks rotation_cohort (needs per-packet timestamps) and any
future `rssi_signature` signal (needs per-packet RSSI per sniffer).
Empty packets table means those signals never have data to score
against.

The fix is small but the *trade-off* is real: enabling packet
writes will grow the DB substantially. Rough estimate: at a
modest 1 K decoded packets/sec average across the sniffer pool,
~50 KB/sec → 4 GB/day. The current 91 M rows mostly come from
`device_ad_history` upserts, which dedup by `(device_id, ad_type,
ad_value)` and don't grow per-packet — so adding `packets`
storage would change the growth model from O(unique-payloads) to
O(packet-count).

Two options:

1. **Off by default + UI toggle.** Add a checkbox to the live-
   capture toolbar ("record packets" / "store full history"), pass
   the flag through to `IngestContext`. User opts in when they want
   to enable rotation_cohort / rssi_signature.

2. **On by default with retention.** Always write to packets but
   add a daily cleanup that deletes packets older than N days
   (cluster runner only needs recent ones anyway). Complexity:
   need a periodic vacuum / DELETE pass.

Recommendation: option 1 for now. Lower-risk; matches the
schema-design intent (the `keep_packets` column literally exists
to gate this). Users who want sub-second cluster signals opt in.

This is the most valuable thing on this list to ship next, because
it unblocks two whole future signal categories — rotation timing
*and* RSSI fingerprinting — by flipping a single bool.

### 2. `bis_access_address_match` [HIGH, ~4 hours]

Solves the C82 case: the periodic-train identity (id 13418, public
MAC) and the primary-channel identity (id 13677, RPA) don't share
Continuity payloads (the C82 isn't an Apple device). They DO share
the BIS access-address fingerprint embedded in the SyncInfo carried
by ADV_EXT_IND's AuxPtr (which 13677 emits) and the AUX_SYNC_IND
(which 13418 emits). The signal needs:

- Store SyncInfo per primary-channel-source device row (new column
  or new table, decoded from AuxPtr at ingest time)
- Store the BIG access address per data-channel-source device row
  (decoded from AUX_SYNC_IND header)
- Match: two devices are the same Auracast source if their access
  addresses match

This is a high-value but moderate-cost addition because it requires
ingest-pipeline changes (parse AuxPtr, parse AUX_SYNC_IND header)
plus a new column/table for the access-address mapping.

### 3. `stable_byte_match` [MEDIUM, ~3 hours]

Catches what `apple_continuity` misses: cross-rotation-epoch RPA
pairs of the same Apple device. Two AirPods broadcasts captured an
hour apart won't share long Continuity payloads (everything has
rotated), but they WILL share the model + state stable bytes from
type 0x07.

Risk: if the user's environment has multiple identical-model AirPods
units, naive stable-byte matching false-positives. Mitigate with a
co-session gate: only match if the two devices appear in the same
sniffer sessions with overlapping observation windows.

Implementation sketch already in `_continuity_protocol.py` — every
TLV's `stable_prefix` field is precomputed. Signal becomes ~50 lines
plus a co-session SQL helper.

### 4. `co_session_packet_ratio` [MEDIUM, ~3 hours]

Solves a weaker version of the C82 case generically. Two device rows
that:

- Share ≥ N sessions
- Have consistent packet-count *ratios* across those sessions
  (within tolerance, e.g. coefficient-of-variation < 0.3)

…are likely two identities of one physical device. The C82 example
was 13 shared sessions with consistent 2.4-3:1 packet ratios — that's
a unique fingerprint independent of any protocol-level analysis.

False-positive risk is real (devices that just sat near the same
sniffers will share sessions). Mitigate by requiring a stable ratio
across many sessions, not just shared presence.

### 5. `rssi_signature` [HIGH, ~6 hours]

The textbook RPA-collapse signal: two RPAs from the same physical
device, observed concurrently by the same sniffers, will have
near-identical RSSI distributions (because they're physically the
same device at the same distance). Implementation:

- Per (device_id, sniffer_id), compute mean/std/percentiles of RSSI
- For each pair of devices observed by overlapping sniffer sets,
  compute a similarity score on their RSSI distributions
- High similarity → same physical device

Expensive both in compute (per-device statistics) and in evidence
accumulation (need many packets per sniffer to get reliable stats).
Highest precision but biggest engineering effort.

## Operational recommendations

### Apply the apple_continuity PR

PR #45 is ready for merge. After merging, restart btviz to reload
profiles. The cluster runner will start producing meaningful merge
output on the next 15-second tick. Verify by tailing the cluster log
and grepping for `merge` lines:

```sh
tail -f ~/.btviz/cluster.log | grep -E '^.*merge'
```

### Adjust the `show:` window for cluster work

The cluster runner's hydrator uses the same `recent_window_s` as the
canvas filter (per the existing wiring). For active cluster
development, **15m or 30m** is the sweet spot:

- Long enough to capture multiple RPA rotation epochs
- Short enough that the O(n²) candidate-pair count stays manageable
  (with `apple_continuity` we got 1.4K pairs for 71 devices in 1h;
  scaling matches O(n²))

For "pure validation" runs, `all` is fine but expect tens of seconds
per tick at 3700 devices.

### Verbose logging when iterating signals

The Verbose cluster log toolbar toggle (added in PR #37) flips the
logger to DEBUG so each abstain produces a line with the partial
signal contributions. Use this when tuning profile thresholds — you
get to see exactly what scored and what didn't for every pair, which
is essential for catching regressions like the rotation_cohort cache
issue.

### Don't use `all-time` clustering output operationally

Even with apple_continuity, the cluster collapse rate at all-time is
1% (39 of 3709 devices absorbed). That's because Continuity payloads
rotate every 15 min, so old RPAs have no shared payloads with newer
ones. The cluster runner is useful for *recent* activity, not for
deduping the historical RPA fleet. The historical fleet collapse
needs the RSSI / stable-byte / co-session signals listed above.

## Active probing — not pursued

The user's note included "you can even initiate connection requests
to discover more data." I deliberately stayed passive for this work
because:

- Connection requests touch real devices in the user's environment
  and could disturb the live capture or alter device-side state
- The right tools for this are interactive (nRF Connect, bluetoothctl)
  rather than scripted
- All findings here come from passive analysis of already-captured
  advertising data — no in-situ modification

If active probing turns out to be useful (e.g. for GATT-level device
discovery to confirm an unknown 0x2720 is AirPods 4 or to read the
Auracast Broadcast Audio Source service), the right shape is a
separate tool that the user runs interactively, not part of the
cluster runner. Worth considering for a future tools/ subdirectory.
